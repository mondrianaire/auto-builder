# Auto Builder Architecture — Operational Documents

Companions to (but distinct from) the conceptual `revised_instructions.md` in the parent folder. Those describe intent and rationale. These describe operations.

## Files

- **`file_schemas.md`** — Every file the system writes, who writes it, what shape it takes, how it's versioned. Directory layout and permission table at the top.
- **`role_charters.md`** — Every role's prompt, written so it can be passed verbatim as the system prompt to a dispatched sub-agent. Includes briefing format, process, output schema, boundaries.
- **`principles.md`** — The four underlying principles (verification fidelity, audit completeness, spec-to-test coverage, path coverage) that the v1.x amendments instantiate. Written to shift the architecture from reactive enumeration of rules to proactive enforcement of laws. Also includes predicted-next-failures by principle. **New v1.6+; consult before proposing any amendment.**
- **`build-lifecycle.md`** — The two-phase model for build delivery (Phase 1 Initial Delivery, Phase 2 In Limbo), the three completion gates (instructions / access / verification), and the mandatory fork-and-archive ceremony when complete. **The architectural boundary between AutoBuilder-as-build-factory and post-build product life — defines what stays in this repo and what forks out to standalone `mondrianaire/{slug}-AB` repos. New 2026-05-15; consult before reasoning about per-build progression.**
- **`failure-catalog-streamdock.md`** — Retrospective on the StreamDock run; seeded the v1.9 principles (E/F/G/H) and the v1.9 Editor role.

Per-run analyses in `runs/{name}/root-cause-analysis.md` examine each prior run through the principles lens — what does this run reveal about which principle was or wasn't being honored. These are companions to `run-report.md` (which is the in-the-moment reflection); root-cause-analysis is the through-line analysis across runs.

## How to Use

To run a build:

1. **Orchestrator** is the entry point. Pass the Orchestrator charter (from `role_charters.md`) as a system prompt to a fresh Claude instance, along with the user's prompt.
2. The Orchestrator creates the project root and file substrate per `file_schemas.md` § Directory Layout, then dispatches Discovery.
3. Each subsequent dispatch follows the same pattern: dispatching role assembles a briefing packet (per the schemas), passes the receiving role's charter as system prompt + the briefing as the user message, and awaits completion.
4. The build runs to convergence; Orchestrator delivers final output to user.

## What These Documents Are *Not*

- Not a software framework. Nothing imports anything. The "agents" are sub-Claudes dispatched via the Cowork Agent tool.
- Not exhaustive. Schemas show required fields; agents may add more. Charters describe load-bearing process; agents use judgment within boundaries.
- Currently v1.10.1 (Sev-4-routing cleanup over v1.10's commit-cadence over v1.9's principles). Expect further revisions as more builds surface new patterns.

- **v1.9** — largest amendment in the project to date, driven by the StreamDock-plugin retrospective (see `failure-catalog-streamdock.md`). The StreamDock build passed every internal gate (Critic clean, CV pass, Orchestrator delivered verdict `pass`) and failed the simplest user-facing test: the plugin did not appear in the host application. Combined with the same first-contact failure in poker 1.0 and the latex CDN failure pattern, this amendment elevates the project's mission to operational rules.

  **North Star** added to `principles.md`: *"The perfect Auto Builder would understand exactly what the user means and build them exactly what they want."* The asymmetric requirement it operationalizes: **prompt specificity must not scale with the architecture's failure modes** — if the architecture needs the user to over-specify to succeed, that's an architecture bug, not a user bug.

  **Four new principles** in `principles.md` (E, F, G, H), each enforced structurally:
  - **E — Atomic Lexical Anchors:** proper nouns in the prompt are atomic identifiers, not descriptive vocabulary. Discovery enumerates them in `ledger-v1.proper_nouns[]` with role (target_defining vs supportive) and `verification_status`; demotion requires four guardrails; lexical markers (`sample`/`example`/etc.) automatically pre-weaken atomicity.
  - **F — External Authority Discipline:** every load-bearing inference about an external system carries a citation with a re-fetchable `verbatim_excerpt`. Training-data familiarity is not a citation. When canonical source is unreachable, the architecture escalates to Discovery Demotion Mode or Sev 4 surfacing — never silently substitutes.
  - **G — Deliverability Tier Discipline:** three explicit verification tiers — Tier 1 telos (PNV, existing), Tier 2 first-contact (new), Tier 3 sub-goal (existing). CV runs Tier 2 first among behavioral checks; Tier 2 failure halts verification immediately. The architecture finally has a structural gate for "can the user reach the artifact at all."
  - **H — Verification Independence:** the verifier and the verified must not share their source of truth. Every assertion has a `source` field (`prompt` / `canonical_evidence` / `td_plan` / `external_source_unreachable`); `td_plan`-sourced assertions on external-system properties are structurally invalid (the StreamDock fake-SDK was modeled on TD's own plan — under v1.9 this is a hard fail flagged by Editor and skipped by CV). CV's `production_fidelity_environment` now requires per-component `real`/`modeled` tagging; modeled components must cite external documentation.

  **Discovery is re-chartered** as the *authoritative intent holder* queryable throughout the build (not just a one-shot ledger producer). New fields on the ledger: `telos` (single-sentence canonical user want), `execution_context_observed` (platform/path evidence Discovery inspected — the StreamDock Windows-paths-but-macOS-default failure is now structurally caught), `proper_nouns[]`, `first_contact_requirements[]`, `importance_action` (forces high-importance IPs to take concrete action: Researcher dispatch / Sev 4 surface / explicit evidence — silent default is a Principle F violation).

  **Discovery (Demotion Mode)** added as a new mode of the existing Discovery role. Invoked when a proper noun whose canonical source was required cannot be verified. Four guardrails (G1 genuinely unreachable / G2 supportive role / G3 telos preserved / G4 substitution satisfies) — all four must hold for the `demote` verdict specifically. **Five outcomes (no `block`):** `demote` (silent substitution + Historian log), `substitute` (TD finds substitute preserving the telos), `best_effort_target_commitment` (Discovery commits to most plausible interpretation when G2 or G4 fails), `rebrief_research` (Researcher re-probes), `insufficient_evidence` (Researcher findings not credible per Principle F). Per the project's North Star contract, the architecture always delivers an artifact — there is no `block` outcome and no user surfacing. Uncertainty is documented in the run-report's Uncertainty Manifest, never raised as a mid-build question. Critic audits demotion records for guardrail completeness and for non-empty `uncertainty_manifest_entry` on `substitute` and `best_effort_target_commitment` verdicts.

  **Editor role** added as the architecture's missing prompt-fidelity auditor. Dispatched by Orchestrator between TD's initial output and Coordinator's first wave. Six structural checks: proper_noun_citation (Principle E + F) / discovery_high_importance_action (Principle F) / td_ip_source (Principle H) / first_contact_coverage (Principle G Tier 2) / telos_coherence. Verdicts: `pass` / `pass_with_recommendations` / `route_to_discovery` / `route_to_td` — there is no `route_to_user`. Editor loop is iteration-capped at 3 passes; after the cap, the architecture commits to current best-effort plan and the build proceeds with unresolved findings in the Uncertainty Manifest. The North Star contract — always deliver — overrides the gate when iteration would otherwise halt the build. Distinct from Critic (substrate consistency post-build) and CV (artifact exercise post-build) — Editor audits prompt fidelity pre-build. Per Addendum C of the failure catalog, Editor's check is structural ("does TD have a citation?"), not substantive ("is the citation correct?"); substantive verification stays with Researcher to avoid re-creating Law B's failure mode where the auditor itself falls back to training-data familiarity.

  **CV charter** updated for Principle H: per-component real/modeled declarations required in `production_fidelity_environment`; modeled components without external citation are hard fails. Process restructured into explicit Tier 2 → Tier 1 → Tier 3 ordering; Tier 2 first-contact runs first and halts pipeline on failure. New `principle_h_skips[]` field records assertions CV refused to verify because they were self-referential against external system properties.

  **Researcher charter** updated for Principle F: every finding requires `citations[]` with `verbatim_excerpt` and `citation_class` (static_canonical / archived_snapshot / interactive_doc). `external_source_unreachable` flag triggers escalation to Discovery Demotion Mode or Sev 4 — never silent fallback.

  **Schemas** updated in `file_schemas.md`: new fields on ledger-v1, new schemas for `decisions/discovery/demotion-v{N}.json` and `decisions/editor/review-v{N}.json`, `source` field added to every assertion type, `citations[].verbatim_excerpt` required on Researcher findings, `production_fidelity_environment.components[]` per-component tagging required on CV reports, `first_contact_results[]` and `principle_h_skips[]` added to CV report schema.

  **Validation strategy:** v1.9 was implemented in response to the StreamDock retrospective specifically because the same prompt can be re-run verbatim as an A/B test of the architecture's defenses (per `feedback_autobuilder_ab_testing_strategy.md`). The build's failure modes (proper-noun loss, OS-platform misread, fake-SDK fidelity gap, first-contact gate missing) are the specific failure modes v1.9 was designed to structurally prevent. The architecture's success on the re-run is the measurement of whether the principles can be enforced as written.

## Suggested First-Time Run

For a first run on the v1.4 architecture, start smaller than blackjack:
- A single-page tic-tac-toe game, OR
- A digital clock with a settings panel

Once one runs end-to-end, attempt more complex prompts with whatever refinements the first run surfaces.
