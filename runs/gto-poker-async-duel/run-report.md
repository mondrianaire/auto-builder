## What the deliverable actually is

A vanilla-JavaScript ES-modules static site, ~20 files under `output/final/`, ready to commit to a GitHub Pages publishing source (repo root, `docs/`, or `gh-pages` branch). Connects at runtime to a Firebase project the user creates on the free Spark plan — Firestore for shared game state and Anonymous Auth for the security-rules predicate that locks each game's documents to its two participants. Push notifications use the W3C Web Push protocol with VAPID, signed client-side and POSTed directly to the recipient's `pushSubscription.endpoint`, bypassing FCM's server-OAuth-required HTTP v1 API. Service worker registered at `./sw.js` with scope `./` so it scopes correctly under a GitHub Pages project subpath. A W3C Web App Manifest with `display:standalone` enables iOS Add-to-Home-Screen push.

Twenty hand-authored No-Limit Hold'em "gotcha" scenarios live in `data/scenarios.json` (the ancestor `gto-poker-trainer` content structure was different enough that direct porting wasn't clean; the new set is authored to the schema TD locked). Player A creates a game with a configurable round count + handful size, gets a share code/URL, sends it to Player B. Each turn, the active player answers a handful, sees a "waiting" screen, and Player B is notified (out-of-app push if opted in, in-app indicator otherwise). After both submit a round, the reveal panel shows each player's action + confidence (1-5) + optional per-scenario note alongside the GTO-verified action. At the configured final round, the wrap-up presents individual performance stats, agreement stats, and an explicit highlighted list of scenarios where actions differed AND both players reported high confidence, ranked by the minimum of the two confidences (descending).

## What broke (or strained)

1. **Coordinator skipped the v1.6 delivery-required Historian artifacts.** Coordinator wrote `state/coordinator/build-complete.json` with `ready_for_final_verification: true` but did NOT write `history/build-summary.md` or `history/decision-index.json` first, even though the Historian Charter explicitly says: *"If you operate inline-collapsed under inline dispatch (the typical case), the Coordinator agent must produce these artifacts as part of its inline-execution duties before writing `state/coordinator/build-complete.json`. Skipping them blocks delivery."* The Orchestrator caught the gap during the delivery-checklist preflight, dispatched Historian as a separate Agent call, and the artifacts landed. No actual delivery delay, but the Coordinator's charter compliance failed silently — Critic's final-sweep didn't catch it because the delivery-checklist preflight is the Orchestrator's responsibility, not Critic's.

   **Refinement question:** should Critic's `final_sweep` checklist include a `delivery_artifacts_present` check (does history/build-summary.md exist? does history/decision-index.json exist?) before signing off? The current gap is that Coordinator's compliance with Historian-collapse is checked by no automated gate — only by Orchestrator's downstream delivery preflight. Adding it to Critic's check list would catch the violation at the substrate-consistency layer rather than at delivery.

2. **Discovery's proper_noun record-keeping is updated by no role after Researcher findings return.** When Probe 001 returned canonical evidence for the `GitHub Pages` proper noun (`verbatim_excerpt` quoting GitHub's docs page on Pages static-only hosting), no role transitioned `proper_nouns[PN.2].verification_status` from `pending` to `verified`. TD's section file references the canonical evidence via `citations_pointer`, so the substantive coverage is intact — but the ledger record itself stays out of sync with the build's actual evidence state. Editor flagged this as F.1 (medium): *"PN.2 verification_status left at pending despite Researcher dispatch + CIT.1 verbatim_excerpt. Substantive compliance intact; status field not transitioned. Discovery record-keeping gap, not TD non-compliance."* Critic's final-sweep agreed it was non-blocking. But it's a small dishonesty in the ledger: the field claims one state and the build's actual state is another.

   **Refinement question:** which role owns updating `proper_nouns[].verification_status` after Researcher findings land? Candidates: (a) TD on receiving findings, as part of locking the IP resolution; (b) Researcher itself, with explicit write authority into the ledger's verification_status field (currently Researcher writes only findings.json); (c) a "Discovery (Receive Findings)" sub-mode dispatched by TD when canonical evidence arrives. The simplest fix is probably (a): TD already references the citations in `citations_pointer`, so adding a one-line `proper_nouns_status_update[]` field on `sections-v1.json` that the ledger reader-of-record can apply as a virtual amendment would keep the writer-permission boundaries clean (TD doesn't write to ledger; the reader composes them at consumption time).

3. **PN.1 ("GTO Poker Training") was demoted inline in the ledger rather than as a separate `demotion-v1.json`.** Discovery (initial mode) included a `demotion_note` field on the proper-noun entry summarizing why PN.1 was supportive — but the formal v1.9 demotion-mode flow (which produces `decisions/discovery/demotion-v{N}.json` with all four guardrails documented explicitly) was not invoked. Editor flagged as F.2 (low): all four guardrails are articulated, just not in the standard schema location. The decision itself (PN.1 names the ancestor build, not the target of this build; telos is expressible without it; the new artifact is independently named) is sound.

   **Refinement question:** should Discovery's initial mode have authority to produce demotion records inline (for the trivial case where a proper noun's demotion is unambiguous at first reading), or should demotion *always* require Researcher findings + four-guardrail check + a separate `demotion-v{N}.json` even when no actual canonical evidence is in question? The architecture currently implies the latter (Demotion Mode is invoked when canonical source is unreachable, which presumes Researcher dispatched and returned empty). For names that are *not* external systems at all (here, the name of a sibling project in the same workspace), the formal demotion flow is over-engineered. A "lightweight demotion" tier might be appropriate: declare the supportive role + role-defining lexical context + the telos-coherence test in the initial ledger entry itself, no separate file required. This would have been the right path here.

4. **PNV.1 cannot be exercised end-to-end live in any verifier environment.** The deliverable connects to a Firebase project the user creates. Until the user creates that project, pastes the config, generates VAPID keys, and pushes to a Pages-enabled repo, there is no live system to exercise PNV.1 against. The CV did the most-honest available thing: it ran static analysis + code-trace + cited the edge-case Node-harness test report (32/32 PASS) as the closest available behavioral evidence, with the firestore_backend/web_push_service components declared `modeled` and pointed at canonical Firebase/W3C docs. This is exactly the Principle H pattern. But it does mean PNV.1's "two browser profiles, real Firebase, real push delivery on a closed tab" scenario is verified *structurally* in the architecture's sandbox and *behaviorally* only when the user deploys. The README acknowledges this honestly under the "what's verified vs deferred" section.

   This is the same shape of issue that v1.5 fixed for latex-equation-renderer (where CDN substitution faked production-fidelity) and v1.9 fixed for StreamDock (where a fake SDK host modeled on TD's own plan faked end-to-end). For BaaS-dependent artifacts, the fix isn't quite the same: the verifier *cannot* run against the user's eventual Firebase project because the verifier and the user are different parties. The honest position the architecture takes — declare the modeled components, cite external docs, ship the artifact with a load-bearing README — is correct in spirit but structurally inferior to a fully live exercise.

   **Refinement question:** should the architecture introduce a "Tier 2.5 / Deferred-Live" verification tier for artifacts whose first-contact requires user-side setup that the verifier cannot impersonate? Right now Tier 2 first-contact says "the user can reach the artifact." For a BaaS-dependent artifact, "reach the artifact" includes "after creating their own backend project." The current verifier exercises every step of Tier 2 except the parts that depend on the user's Firebase project existing. A Deferred-Live tier could explicitly catalogue what's verified-now vs verified-on-first-deploy and surface that to the user as a structured checklist instead of as README prose. The user's README receipt would explicitly say "items 1-6 are verified; items 7-9 will be verified by your first deploy." This would also help future runs whose prompts imply BaaS-style deliverables (Auth0, Stripe, Twilio, etc., all have the same shape).

5. **Deliverability/setup-burden tension on "fully can be hosted on github pages".** The user said "It is imperative that this application fully can be hosted on github pages." The strict-literal reading would forbid any third-party service dependency. The substantively-correct reading (which Editor explicitly endorsed in its `pass_with_recommendations` rationale) is that the deliverable IS hosted entirely on GitHub Pages — the Firebase project is a service the app calls, not a server the user self-hosts. By that reading, every SPA that calls a third-party API is "fully hosted on GitHub Pages." This is honest, but it pushes the setup burden onto the user: create Firebase project, enable Firestore + Anonymous Auth, paste config, generate VAPID, paste rules, enable Pages, optionally Add to Home Screen on iOS. Six to ten steps depending on platform.

   The alternative architectures considered (URL/code-payload exchange with the user copying a serialized game state every turn; Gist API as backend with anonymous-write deprecation problems; pure WebRTC with synchronicity constraint) all violated something the user named. The Firebase path was the best fit, but the cost is the setup burden. The README owns this honestly. But there is a real reading of the prompt under which the user expected to download the deliverable, drop it into a repo, enable Pages, and play with their mom — no Firebase. Under that reading, the build doesn't satisfy intent.

   **Refinement question:** when the prompt's literal constraint and the prompt's stated goals are in tension, and Discovery + TD + Editor + Researcher all converge on the same compromise, the architecture currently commits to that compromise and surfaces the uncertainty in the run-report's "Uncertainty Manifest." That's the right move per the v1.9 North Star contract (always deliver; never halt to ask). But the uncertainty is large enough here that the *user* would probably want to know this before reading 1,000 lines of README. Should the architecture's "delivery surface to user" include a one-paragraph "what you'll need to do to actually use this" preamble that precedes the file pointer, written by the Orchestrator from the run-report's setup-burden flags? The README already has this content; the question is whether the delivery message itself should also.

6. **Inline-mode Researcher: the TD agent inlined Probe 001 into its own execution rather than dispatching a real sub-agent.** TD's report explicitly noted: *"No subagent/Agent tool was available in this environment, so I performed Probe 001's research inline via WebSearch/web_fetch, captured 6 verbatim canonical citations, and wrote both `briefing.json` and `findings.json` to `research/probes/probe-001/`."* This is operationally fine — the citations are real, the verbatim_excerpts are populated, and the findings.json is in the canonical location for downstream consumption. But the v1.9 Editor charter's `proper_noun_citation` check relies on the assumption that findings come from an *independent* Researcher dispatch (Principle H — verifier and verified must not share their source of truth). When TD inlines the Researcher into its own context, the source-independence is weaker than the architecture assumes: TD is reading the docs, TD is locking the IP resolution, TD is writing the findings.json, and Editor is checking TD's own writeup for citation presence.

   **Refinement question:** is "TD inlines Researcher into its own execution" a documented inline-deviation category, or is it implicit in the inline-dispatch mode for the entire run? Currently only Coordinator's inline-mode is explicitly chartered (Coordinator-as-Overseer-as-Builder). TD's charter doesn't enumerate an inline-Researcher path. Either: (a) add Researcher to the inline-deviations category list (with `nested_equivalent: "a real Researcher sub-agent would have been dispatched"`); or (b) charter TD to always nest-dispatch Researchers as real sub-agents, even when the rest of the run is inline-mode at Coordinator level. Option (b) is structurally cleaner for Principle H. Option (a) is operationally honest about what happens in practice.

## Refinement candidates for the architecture (consolidated)

Numbered for triage. None of these block the run; all are visible in what happened.

1. **Add `delivery_artifacts_present` to Critic's `final_sweep` checklist.** Catches Coordinator's silent failure to write `history/build-summary.md` + `history/decision-index.json` before signaling build-complete.

5. **Charter the `delivery surface to user` step in the Orchestrator to include a "what you'll need to do" preamble** for builds that flagged setup burden. Discovery's restatement and the run-report's uncertainty manifest already have the content; the delivery message should index it.

6. **Decide whether TD's inline-Researcher path needs a structured inline-deviation log entry.** Either add a Researcher category to the existing inline-deviations list, or charter TD to always nest-dispatch Researchers for Principle H source-independence.
