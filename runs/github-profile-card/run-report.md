# Run Report: github-profile-card

**Prompt:** "Build me a tool that takes a GitHub username and shows me a developer profile card for them — their pinned repos, their contribution streak, their most-used language, and a small visualization of their commit activity for the last 90 days."
**Date:** 2026-05-16
**Run root:** `runs/github-profile-card/`
**Final artifact:** `runs/github-profile-card/output/final/index.html` (+ `css/`, `js/`, `README-RUN.md`)
**CV verdict:** `pass_with_concerns` (10/10 first-contact requirements pass; all caveats trace to the user-supplied PAT, intrinsic to the design)
**Edge-case-testing:** 77 assertions, 67 pass / 0 fail / 10 correctly deferred to CV Tier 2
**Total Orchestrator-spawned agent dispatches:** 7 (Discovery → TD → Editor → Coordinator → Critic-final-sweep → CV → Delivery)

## Outcome

The build delivered a single-page, dependency-free GitHub Profile Card lookup tool: an `index.html` shell plus CSS, three JS modules (`api-client`, `data-derivers`, `card-renderer`), and a `main.js` wiring file. The user types a GitHub handle, pastes a Personal Access Token, and gets back a card showing the avatar/handle/bio, three stat blocks (current streak, most-used language, total contributions for the year), up to six pinned repositories, and a 91-cell heatmap captioned "Contribution activity (last 90 days)". Error paths (user-not-found, bad PAT, rate-limit, network failure, no PAT) all surface as actionable status-region messages rather than blank states. The page is served from a tiny local static server (Python `http.server` or `npx serve`) because ES module loading over `file://` is blocked by modern browsers; this is documented in `README-RUN.md`.

CV verified the artifact end-to-end under jsdom against canonical-evidence-shaped fixtures derived from GitHub's documented API response schemas. The render path produces all five named card elements; all five typed error classes map to their intended UI messages; the repeat-lookup path cleanly replaces the prior card. The two `pass_with_caveat` first-contact items (FC.1 console-cleanliness on a real browser engine; FC.4 live `api.github.com` round-trip with a user PAT) are both Tier-2-deferred by the CV charter — the user's first contact is the moment of live-network truth.

## What worked

- **Discovery's evidence-backed IP1 resolution paid off twice.** IP1 (auth/PAT) was the only high-importance inflection point, and Discovery resolved it `evidence_backed` inline with explicit citations to GitHub's GraphQL auth requirements and REST rate-limit documentation. That single decision (a) determined the entire downstream architecture (GraphQL for `pinnedItems` + `contributionsCollection`; REST for basic user + per-repo languages), (b) supplied the canonical-evidence source for every external-system MCA that TD later wrote, and (c) gave CV a defensible substrate for Principle-H modeling when the live PAT round-trip was unavailable. Principle E (Decision Grounding) fired exactly as intended — no platform misread, no API shape guessed.
- **Principle F (Coverage) held across 89 of 89 traceable items.** All 11 ledger assumptions, all 12 OOS items, and all 6 IP default branches were verified by CV. The 47 Discovery Coverage Assertions (DCAs) sit alongside 22 Machine-Checkable Assertions (MCAs) for a total of 69 coverage edges plus the 8-section/edge-case structure. The few coverage hygiene gaps the Critic raised are graph-walk artifacts, not missing-test gaps (see "What broke" below).
- **Principle G (Honest Labeling) showed up where the prompt's noun ambiguity could have bitten.** "Streak" → `Current streak: N days` label. "Most-used language" → `Most-used language across public repos`. "Commit activity for the last 90 days" → caption `Contribution activity (last 90 days)` (honest under the chosen GraphQL data source, which includes PRs/issues/reviews alongside commits). First-contact-comprehension defects were prevented by labeling, not by data-source acrobatics.
- **Inline mode was the right call for 5 sections.** Coordinator handled Overseer + Builder + Integrator work for all 5 sections directly; wave 1 had genuine 2-way parallelism (api-client ‖ data-derivers, no inter-deps), waves 2–4 sequenced cleanly. Total Coordinator wall time was ~8 minutes for the entire build phase. Two inline deviations were logged with their nested-dispatch equivalents; both stayed inside the legal scope categories.
- **Contracts paid off at integration with zero glue code.** `manifest.json` reports `glue_code_added: []` and `issues_resolved_during_integration: []`. The four original contracts pinned the function signatures (`fetchProfilePayload(username, pat) → ProfilePayload`), the five error class names, the GraphQL body shape, and the Authorization header form. By the time the Integrator ran, every import path and every DOM ID lined up because the contracts said so. The ui-shell's `import` lines (`./api-client.js`, `./data-derivers.js`, `./card-renderer.js`) matched the api-client/data-derivers/card-renderer file names without coordination overhead.
- **No-Researcher-but-evidence-backed pattern.** Discovery's IP1 gathered the canonical GitHub docs inline; TD then captured those same references in `research/probes/probe-quick-github-api/findings.json` as a citations file so every MCA touching the GitHub API could source as `canonical_evidence` with a `citations_pointer`. This INLINE-mode alternate-route verification path saved a Researcher dispatch without losing the audit trail — every external-system claim is still grounded in a citable document.
- **Zero escalations end-to-end.** Editor returned `pass_with_recommendations` (two low-severity housekeeping items, no routing). Critic returned three findings (two low, one medium), none requiring an escalation packet. CV returned `pass_with_concerns` with caveats that don't block delivery. The architecture's always-deliver contract held cleanly on a happy path.

## What broke / what was surprising

1. **Critic's medium finding on contract coverage edges.** The most consequential audit hit: `contracts/original/*.json` `interface` and `notes` fields are coverage-required per `file_schemas.md`, but no assertion's `covers` field points at any contract-path target. Every contract export *is* exercised by an implementation-side MCA (api-client signatures, error names, GraphQL body shape, Authorization header form), so coverage is in place in *practice* — but under a strict graph walk the contracts read as "uncovered". This is the cleanest example yet of file_schemas' coverage-required table outpacing the assertion-graph semantics. Flagged for v1.10+ amendment loop (see Refinements below).
2. **14 CV `pass_with_caveat` results all share one root cause.** The intrinsic PAT-user-side design from IP1 means CV cannot exercise the live `api.github.com` round-trip without a user-supplied token. CV correctly modeled this under Principle H using octocat-shaped fixtures derived from canonical evidence, but the caveat appears on every assertion that crosses the network boundary (MCA.api-client.2/3/4, DCA.A2/A6/PN5/OOS.orgs, DCA.FC1/FC4, PNV.1, MCA.edge-case-testing.2, MCA.card-renderer.1 transitively, etc.). The verdict format (`pass_with_caveat` per assertion) makes the audit read noisier than the underlying truth — there's really one caveat repeated fourteen times.
3. **Two source-grep false positives in initial edge-case-testing.** `\blogin\b` matched GitHub's own API field name for username (`user.login`, `repo.owner.login`) throughout legitimate code; `@media` matched a CSS comment that documents the OOS choice. dev-002 refined the patterns to match the *intent* (auth-flow constructs / `@media` followed by a media-query paren) rather than bare substrings. Surprising not because the false positive happened, but because static-checks-as-source-grep is structurally vulnerable to vocabulary collisions whenever the build's domain uses words that also name forbidden patterns. The fix worked; the class of issue is worth flagging.
4. **Two low-severity Editor recommendations didn't escalate but did expose a hygiene shape.** F.1: ledger PN.1–PN.6 still carry `verification_status: pending` even though TD covered them via `canonical_evidence` + `citations_pointer`. F.2: DCA.PN2 source could be upgraded from `td_plan` to `canonical_evidence`. Neither is build-breaking, but both point at the same thing — Discovery → TD verification handoff doesn't currently produce a "verified_by_alternate_route" annotation back on the ledger. The proper-noun verification trail is forward-only.

Nothing was strictly *broken* — the build passed every gate. The findings above are the architectural awareness items the run surfaced.

## Uncertainty Manifest

Per the v1.9 contract, the guesses this build committed to:

- **PAT-required design (IP1) is the only acceptable path.** GraphQL `pinnedItems` and `contributionsCollection` cannot be reached unauthenticated; bundling a token was rejected; therefore the user must obtain and paste a PAT. Mitigation: prominent labeled `pat-input` field, explanatory note, anchor link to `github.com/settings/tokens/new` with the `read:user` scope hint. Failure mode if wrong: user can't see pinned/contribution data and gets the degraded-path message instead of a working card.
- **Streak = current streak (A6/IP2).** Trailing consecutive-positive-day count ending at the most recent positive day. Labeled `Current streak: N days` on the card so the metric self-discloses. Failure mode if wrong: number shown doesn't match the user's mental model (e.g., they expected longest streak); mitigated by the label.
- **Most-used language = bytes-weighted across owned non-fork repos (A7/IP3).** Linguist method, matching GitHub's own profile sidebar; forks excluded; tie-break alphabetical ascending. Labeled `Most-used language across public repos` on the card. Failure mode if wrong: reported language differs from user expectation (commit-weighted or repo-count-weighted would give different answers for some users).
- **90-day chart = GraphQL contributionCalendar slice (A8/IP4).** Daily totalContributions over the trailing 90 days, dominated by commits for most active users but inclusive of PRs/issues/reviews. Labeled `Contribution activity (last 90 days)` to be honest about what the bars represent. Failure mode if wrong: user expecting strictly-commits will see slightly higher counts than the events-API path would show.
- **Card form = single-target, dark theme, heatmap viz (A4/IP5/IP6).** One card per lookup; GitHub-native dark palette; 13×7 SVG heatmap with 5-level green intensity. No theme toggle (low-importance IP, deferred). Failure mode if wrong: a user wanting a light-theme card or a bar-chart visualization is mismatched; aesthetic preference rather than functional defect.
- **Per-repo /languages fan-out capped at 30 (dev-001).** Most-used-language is computed from at most the 30-most-recently-updated non-fork owned repos. Failure mode if wrong: a user whose dominant language only appears in their 40th-most-recent repo gets a wrong answer. Saturation is high in practice — the top language is stable inside the first 30 for the overwhelming majority of accounts.
- **Editor's `pass_with_recommendations` findings stand.** PN.1–PN.6 carry `verification_status: pending` despite TD's canonical-evidence coverage; DCA.PN2 sources as `td_plan` rather than `canonical_evidence`. Both are forward-only hygiene gaps in the verification handoff, not functional defects.
- **Critic's medium finding (contracts lack direct `covers` edges) stands.** No assertion `covers` field points at `contracts/original/*.json#interface` or `#notes`. Every contract export is exercised by an implementation-side MCA in practice; the graph-walk reads as "contracts uncovered". Architectural awareness item, deferred to v1.10+.
- **CV's production-fidelity caveat.** Live `api.github.com` round-trip not exercised in CV (the PAT is intrinsically user-side per IP1's evidence-backed default). Render path verified end-to-end against canonical-evidence-shaped fixtures under jsdom; the moment of live-network truth is the user's first contact. Console-cleanliness on a real Chromium-class engine also deferred to user first-contact (jsdom parse is clean but Node-based DOM cannot stand in for a browser).

## Refinements to consider

In rough order of architectural priority:

1. **`file_schemas.md` Coverage-Required table needs to reconcile with `covers` semantics for contract files.** Two viable shapes: (a) extend `covers` to permit contract-path targets (`contracts/original/api-client--data-derivers.json#interface.fetchProfilePayload`) and have TD emit one MCA per contract export pointing at it; or (b) add explicit text saying contract `interface`/`notes` are coverage-discharged when at least one MCA passes against each declared export on the implementation side. This was Critic's medium finding and is the most consequential audit gap in the build.
2. **CV's `production_fidelity_environment.components[]` should formalize `user_supplied_at_runtime` as a recognized status.** This run's CV report borrowed the status ad-hoc for the PAT credential component. For any future build where credentials are intrinsically user-side (PAT, API key, OAuth refresh token, locally-installed binary path), a recognized component status would let CV produce a cleaner audit trail than the current Principle-H prose justification — and would distinguish "we couldn't test this because we don't have it" from "we couldn't test this because the artifact is broken".
3. **The INLINE no-Researcher-but-evidence-backed pattern is worth formalizing as a recognized verification path.** This build skipped Researcher because (a) Discovery resolved the only high-importance IP `evidence_backed` inline with citations and (b) TD captured those citations as a probe findings file that every external-system MCA could source as `canonical_evidence`. That's a legitimate alternate route — the audit trail is intact, every external claim is citable — but the Editor's structural audit currently has no field for "Researcher was skipped because the inline citations file substitutes". Either the Editor recognizes this explicitly, or the architecture documents it as a permitted INLINE-mode pattern with the conditions under which it's legal (single-high-IP-with-evidence + inline-citations-file present + all external MCAs cite it).
4. **Carry-forward from prior runs:** mid-build Critic in inline mode (still untested under stress); proper-noun verification status update (forward-only handoff — Editor's F.1 finding); coverage hygiene for per-section `out_of_scope[]` items (Critic finding #2, low severity).
5. **Source-grep static checks need a vocabulary-collision-avoidance pattern.** dev-002 caught two collisions; future builds that target GitHub-related or auth-related domains will hit them again. A reusable convention (test for the *construct* — `@media (`, `signIn(`, `oauth_redirect_uri =` — rather than the bare token) would prevent the same false positives across runs.
